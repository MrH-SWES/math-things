# Math Things Interaction Doctrine

> The obvious action should be the mathematically useful action.

---

## Core Principles

### Object First

Every Math Thing starts with a physical mathematical object. The object is the app. Controls, labels, equations, and records exist to support the object — never to replace it.

### Direct Manipulation Over Typed Input

Students should touch, drag, pull, drop, snap, and move the mathematical object. Typed input is a last resort.

### Physical Constraints Over Answer Checking

The interface should make wrong moves impossible, incomplete, or visibly unstable. It should not accept an answer and then tell the student whether it was right.

### Motion Teaches

Motion should communicate fit, resistance, transformation, equivalence, scale, grouping, fusion, mass, and settle. Animation is not decoration — it is instruction.

### One Clear Mathematical Object Per App

Prefer one object over many representations at once. If an app tries to do too much, split it into smaller tools.

### Symbols Follow Action

The equation is the shadow of the object. Labels, equations, and algorithms may appear after or alongside the action, but they should be optional, delayed, teacher-toggleable, or visually subordinate.

### Teacher-Facing Language Lives Outside the Student Surface

Lesson plans, standards alignment, prompts, and discussion moves live around the app — not inside it. The student interface should stay quiet.

### Student-Facing Mode Controls Should Be Minimal and Preferably Wordless

If mode controls exist on the student surface, they should be iconic, gestural, or structurally obvious. They should not require reading a label to understand.

### No Badges, Points, Levels, or Game Mechanics Unless They Preserve Mathematical Structure

Extrinsic reward systems are prohibited unless the reward system itself encodes the mathematics.

### Equations and Records Should Follow the Object, Not Drive It

Records may update as the student acts. They should never be the primary interface through which students direct the object.

## Correct vs. Incorrect Structure

| Correct structure | Incorrect structure |
|---|---|
| Snaps | Resists |
| Fits | Fails visibly |
| Fuses | Stays incomplete |
| Aligns | Remains unmerged |
| Reveals | — |
| Settles | — |
| Glides | — |
| Locks | — |
| Transforms | — |

---

## Full Interaction Rules

1. The obvious action should be the mathematically useful action.
2. Correct structure should snap, fit, fuse, align, reveal, or settle.
3. Incorrect structure should resist, fail visibly, or remain incomplete.
4. Avoid text explanations inside the student interface.
5. Use labels only when they are part of the mathematical object, not directions.
6. Make symbols optional, delayed, teacher-toggleable, or secondary when possible.
7. Prefer direct manipulation over typed input.
8. Prefer physical constraints over answer checking.
9. Prefer one clear mathematical object over many representations at once.
10. If the app tries to do too much, split it into smaller tools.
11. Student-facing mode controls should be minimal and preferably wordless.
12. Gesture controls should feel like manipulating the object, not operating a dashboard.
