# Math Things App Audit Rubric

Use this rubric before calling any app finished, before recommending it for the catalog, and before submitting any major improvement.

---

## The Final Test

Ask every question before declaring an app ready:

1. **What is the mathematical object?**
   Can you name one concrete physical thing the student is working with?

2. **What is one?**
   Is the unit of measure physically defined and unambiguous?

3. **What action does the student take?**
   Is the primary interaction a physical manipulation of the object?

4. **What structure does the action reveal?**
   Does the action expose a mathematical relationship, pattern, or constraint?

5. **Can the student succeed without engaging the intended math?**
   If yes, the constraints are insufficient. Redesign them.

6. **Is instruction text doing work the interface should do?**
   If a label or tooltip is explaining what to do, ask why the interface isn't making it obvious.

7. **Does the object feel physical enough?**
   Does the rendering, weight, and material quality match the math it represents?

8. **Does motion clarify the math?**
   Does each animation communicate structure, transformation, grouping, or fit?

9. **Does the record follow the object?**
   Are equations and records generated by the student's physical actions, not the other way around?

10. **Could this be simpler?**
    Is there anything that could be removed without losing mathematical meaning?

11. **Does the app stay in its lane?**
    Does it do one thing well, or does it try to be a lesson, quiz, dashboard, or tutor?

---

## Audit Outcomes

### Keep

The tool already enforces the standard cleanly and should remain in the core catalog. No structural changes needed. Minor polish only.

### Spin / Modify

The mechanic is promising but needs a variant, constraint change, or narrower use case. The core idea is sound; the scope or presentation needs adjustment.

### Rework

The concept points at the right standard, but the present design needs substantial restructuring. The math target is correct; the implementation does not yet enforce it.

### New Build

The current tool does not meaningfully enforce the standard. The concept or implementation is too far from the doctrine to be repaired incrementally. Build from scratch.

## How to Run an Audit

1. Read `docs/00_MATH_THINGS_DESIGN_DOCTRINE.md`.
2. Open the app and interact with it as a student.
3. Fill out `harness/APP_AUDIT_TEMPLATE.md` for the app.
4. Assign an outcome: Keep, Spin/Modify, Rework, or New Build.
5. Write a recommended next task and acceptance tests.
6. Save the audit as a new file in `harness/reviews/` or as the latest review.
